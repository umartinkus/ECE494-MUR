# Repository Guidelines

## Project Structure & Module Organization
This is a ROS 2 workspace. Core layout:
- `src/`: ROS 2 packages. Current packages: `esp32_sensors` and `groundstation_interface`.
- `src/esp32_sensors/include/`: public C++ headers (e.g., `marlin_comms/*.hpp`).
- `src/esp32_sensors/src/`: C++ nodes (e.g., `sensor_node.cpp`).
- `build/`, `install/`, `log/`: generated by `colcon` (do not edit by hand).

## Build, Test, and Development Commands
Run these from the workspace root:
- `colcon build`: build all packages in `src/`.
- `colcon build --packages-select esp32_sensors`: build a single package.
- `colcon test`: run tests and linters for all packages.
- `colcon test --packages-select esp32_sensors`: test a single package.
- `colcon test-result --verbose`: show detailed test output.

## Coding Style & Naming Conventions
- Language: C++ for ROS 2 nodes and libraries.
- Compiler flags: `-Wall -Wextra -Wpedantic` are enabled for GCC/Clang.
- Indentation: follow existing file style (typically 2 spaces in CMake, 2–4 in C++).
- Naming: ROS 2 packages and executables use `lower_snake_case` (e.g., `esp32_sensors`, `sensor_node`).
- Headers belong in `include/<package_name>/` and source in `src/`.

## Testing Guidelines
- Framework: `ament_lint_auto` and `ament_lint_common` (lint-focused).
- Add new tests under package `test/` directories if/when needed.
- Keep tests package-scoped; run with `colcon test --packages-select <pkg>`.

## Commit & Pull Request Guidelines
- Git history shows short, informal, lowercase messages (e.g., “adding a new test file”); no strict convention is enforced.
- Recommended: write clear, imperative summaries and include context in the body if behavior changes.
- PRs should include: summary of changes, how to test (`colcon test ...`), and any relevant logs or screenshots.

## Security & Configuration Tips
- Do not commit `build/`, `install/`, or `log/`.
- Use ROS 2 environment setup before running nodes: `source install/setup.bash`.
